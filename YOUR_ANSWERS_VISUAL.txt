QUESTION 1: "What happens kung low ang balance pero may tumawag sa referral?"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When REFERRAL_KEY balance is low:

    Before Fix:                          After Fix:
    ───────────                          ──────────
    
    Low balance                          Low balance
        ↓                                    ↓
    Marked pending                       Marked pending_disbursed ✓
        ↓                                    ↓
    Admin tops up                        Admin tops up
        ↓                                    ↓
    Retry looks for 'pending'            Retry looks for 'pending' + 
    ✗ Not found (wrong status!)          'pending_disbursed' ✓
        ↓                                    ↓
    ✗ STUCK FOREVER                      ✓ AUTO-RETRIES
        ↓                                    ↓
    Users never get G$                   Users get G$ ✓

Status Change Flow:
                                     
    pending_disbursed ← Low balance detected
         ↓
         ↓ (Admin tops up balance)
         ↓
         ↓ (Admin calls: POST /api/admin/referral/process-pending)
         ↓
    completed ← Disbursement succeeds
    
User Timeline:

    Time 0:00  │ User A refers User B → Faces verified → Disburse!
               │ ✗ FAIL: REFERRAL_KEY only has 800 G$, need 1500 G$
               │ Status: pending_disbursed
               │
    Time 0:05  │ Admin sees alert: 45 pending, 45,000 G$ owed
               │
    Time 0:10  │ Admin tops up: +50,000 G$ to REFERRAL_KEY
               │ New balance: 50,800 G$
               │
    Time 0:15  │ Admin clicks: "Process Pending"
               │ System retries all 45 users
               │ ✓ ALL SUCCEED
               │
    Time 0:20  │ User A gets notified: +1000 G$ ✓
               │ User B gets notified: +500 G$ ✓

QUESTION 2: "Si referral key lang naman mag bayad ng gas fee sa disbursement?"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

YES - REFERRAL_KEY pays 100% of gas fees!

When REFERRAL_KEY sends 1000 G$ to User:

    REFERRAL_KEY Wallet                  User's Wallet
    ─────────────────────                ─────────────
    
    Balance: 5000 G$                      Balance: 0 G$
        │
        ├─ Send 1000 G$ to User    →        Receives: 1000 G$ ✓
        │
        ├─ Pay gas fee: 0.001 G$   →        (NO deduction)
        │  (REFERRAL_KEY pays!)
        │
        └─ Total cost: 1000.001 G$
        
    Balance: 3999.999 G$                 Balance: 1000 G$

Who Pays?
┌─────────────────────────────────────┐
│  REFERRAL_KEY                       │ ← Sends the G$
│  REFERRAL_KEY                       │ ← Pays the gas
│  Users                              │ ← Get full amount
└─────────────────────────────────────┘

Real Numbers (1000 disbursements):

    Cost Breakdown:
    ├─ G$ sent to users:       1,500,000 G$
    ├─ Gas fees:                    1 G$ (tiny!)
    └─ Total from REFERRAL_KEY: 1,500,001 G$

    Balance Needed:
    ├─ Minimum safe: 500 G$ (can disburse ~250 users)
    ├─ Normal:       2000 G$ (can disburse ~1000 users)
    └─ Enterprise:   5000+ G$ (run continuously)

Why REFERRAL_KEY Pays:

    Blockchain Rule:
    "The account that SENDS the transaction pays the gas fee"
    
    ┌─────────────────────┐     ┌──────────────────┐
    │ REFERRAL_KEY        │────→│ User's Wallet    │
    │                     │     │                  │
    │ Sends: 1000 G$      │     │ Receives: 1000 G$│
    │ Pays gas: 0.001 G$  │     │ Pays: 0 G$ ✓     │
    │                     │     │                  │
    └─────────────────────┘     └──────────────────┘

Gas Fee Variability:

    Network State      Gas Price    Fee Per Transfer
    ─────────────────────────────────────────────────
    Low traffic        0.1 Gwei     ~0.000025 CELO
    Normal traffic     1.0 Gwei     ~0.00025 CELO
    High traffic       5.0 Gwei     ~0.00125 CELO
    Congested          10 Gwei      ~0.0025 CELO

    But total gas for 1000 users? Only ~1 G$ total!
